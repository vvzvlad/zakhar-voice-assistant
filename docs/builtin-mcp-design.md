# Встроенные MCP-серверы (погода, календарь) — исследование

Как отдать «встроенные» интеграции (погода через ключ OWM, календарь через креды аккаунта)
тем же механизмом, что и внешний умный дом — то есть **как MCP-инструменты**, не городя для каждой
б��споку. Это дизайн-исследование, не реализация.

Связано: [settings-storage-design.md](settings-storage-design.md) (плагины/конфиг),
[web-panel-brief.md](web-panel-brief.md) (там уже была хотелка «несколько MCP + промпт на каждый»).

---

## 1. Сейчас

- `src/mcp_client.py::McpToolHub` подключается к ОДНОМУ внешнему MCP-серверу по streamable-HTTP
  (`streamablehttp_client` + `ClientSession`), листит инструменты, отдаёт их LLM (`hub.tools`),
  исполняет вызовы (`hub.call(name, args)`). Сессии короткоживущие на операцию (само-восстановление).
- Погода — НЕ инструмент: `build_system_prompt` зовёт `get_weather_summary` и **всегда** вшивает
  строку погоды в системный промпт.
- Календаря нет.

Хочется: погода и календарь становятся инструментами, доступными LLM через тот же tool-loop, но это
**внутренние, в коде** интеграции (не внешний контейнер).

---

## 2. Что умеет `mcp` SDK (1.27.2) для in-process

Главное (проверено по докам SDK; точные сигнатуры/возврат свериться при реализации):

- `from mcp.server.fastmcp import FastMCP` — строим маленький сервер декораторами `@mcp.tool()`.
  Схема инструмента (`inputSchema`) и описание генерируются из тайп-хинтов + докстринга **автоматически**.
- У инстанса `FastMCP` есть **прямые корутины** `await mcp.list_tools()` и
  `await mcp.call_tool(name, arguments)` — вызываются прямо в процессе, **без сети, без stdio, без
  `ClientSession`**. Это и есть «маленький MCP-сервер как объект в памяти».
- Есть и in-memory транспорт (`mcp.shared.memory.create_connected_server_and_client_session` /
  `create_client_server_memory_streams`) — даёт настоящий `ClientSession` к in-process серверу. Но в
  одном процессе это лишняя обвязка (task group, lifecycle) без выгоды.

Вывод: встроенный сервер = реальный `FastMCP`-инстанс, к которому ходим напрямую методами. При этом он
остаётся «честным» MCP-сервером — позже его можно вынести в отдельный процесс (`mcp.run(stdio/http)`)
без изменений у потребителя.

## 3. Три подхода и выбор

**A. Внешние процессы/контейнеры (как умный дом сейчас): каждый built-in — отдельный HTTP-MCP.**
Максимальная изоляция и единообразие, но операционно тяжело (лишние процессы/порты/деплой) ради
погоды и календаря. Оверкилл.

**B. Фейковый «MCP-образный» реестр функций** (просто список tool-dict'ов + `call`), без реального MCP.
Проще всего, но теряем «настоящий MCP-сервер, выносимый наружу позже» — а это прямо то, что просили
(«нужны маленькие mcp сервера»). Отклоняю.

**C. (рекомендация) In-repo `FastMCP`-серверы, дёргаемые в процессе напрямую, за общим адаптером
`ToolSource`.** Каждый built-in — настоящий `FastMCP` с `@mcp.tool()`, но в рантайме мы зовём его
`list_tools()`/`call_tool()` без транспорта. Хаб становится мульти-сорсным: внешний HTTP-сервер и
встроенные серверы прячутся за одним интерфейсом.

Почему C: LLM видит единый список инструментов (никакого спец-кода «погода в промпт»); built-in —
честный MCP (выносим наружу позже сменой фабрики, без правок tool-loop); ложится в плагинную
философию (добавить built-in = один файл, ноль правок ядра).

---

## 4. Архитектура: мульти-сорсный хаб

Ввести тонкий интерфейс источника инструментов (нормализует всё к одному виду):

- `ToolSource.list_tools() -> list[tool-dict]` (формат уже есть: `McpToolHub._to_groq_tool`)
- `ToolSource.call(name, args) -> str`

Две реализации:
- `HttpMcpSource` — нынешний `streamablehttp_client` + `ClientSession`, короткоживущие сессии
  (само-восстановление для флапающего внешнего сервера) — по сути текущий `McpToolHub`.
- `BuiltinMcpSource` — обёртка над `FastMCP`-инстансом: `list_tools()`/`call_tool()` напрямую,
  без сессии, всегда доступен (нет сети). `call_tool` возвращает контент-блоки → достаём `.text`
  (нормализация в адаптере, переживёт разницу версий SDK).

`ToolHub` агрегирует N источников:
- `tools` = объединённый список (для LLM);
- `call(name, args)` = маршрутизация во владеющий источник;
- изоляция отказов: один источник упал ≠ остальные упали (как сейчас MCP-аутаж не роняет ассистента);
- `ensure_tools()` рефрешит (built-in статичны, http перечитывает).

**Неймспейсинг.** Имена инструментов префиксовать id источника (`weather__get_current`,
`home__set_light`), чтобы (а) не было коллизий, (б) хаб понимал, кому маршрутизировать `call`. Префикс
снимается перед вызовом конкретного источника. LLM видит префиксованные имена — это нормально.

`src/llm.py` tool-loop НЕ меняется: он и так берёт `hub.tools` и зовёт `hub.call` — нужно лишь
расширить `McpToolHub` до мульти-сорса (или ввести `ToolHub`).

## 5. Интеграция в конфиг/плагины (как добавлять — лучший способ)

Ровно та же плагинная модель, что для стадий, но категория **`mcp` — мульти-инстансная** (в дизайне
панели это уже заложено). Каждый «вид» MCP — in-repo плагин в `src/plugins/mcp/`:

- `http` — внешний сервер: ConfigModel `{url, token}`.
- `weather` — built-in: ConfigModel `{owm_api_key, default_city, units, lang}`.
- `calendar` — built-in: ConfigModel с кредами аккаунта (см. §6).

Каждый плагин даёт: свою `ConfigModel`, флаг `enabled`, опциональный `prompt` (подсказка модели про
эти инструменты — реализует «кастомный промпт на сервер» из брифа панели), и фабрику `create(cfg, deps)
-> ToolSource`. Для built-in фабрика строит `FastMCP` (инструменты + креды из cfg) и заворачивает в
`BuiltinMcpSource`; для http — `HttpMcpSource`.

Хранение в `config.json`: `mcp` — список инстансов `[{instance_id, plugin, enabled, prompt, values}]`.
На старте `app` строит `ToolHub` из всех `enabled` инстансов (фабрики получают общий `deps` с
http-клиентами — built-in погода/календарь ходят наружу через **проксируемый** `http_cloud`).

Итог «как добавить новый built-in MCP»: положить `src/plugins/mcp/<kind>.py` (ConfigModel + FastMCP с
`@mcp.tool()` + `create()`), зарегистрировать. Ноль правок ядра, хаба и LLM-цикла. Появляется в панели
со своей формой настроек автоматически.

## 6. Конкретно погода и календарь

### Погода (OWM) — простой случай
- Built-in `FastMCP` с инструментом `get_current_weather(city: str | None)` поверх существующего
  `get_weather_summary` (логика уже есть). Креды: `owm_api_key` + `default_city` из cfg, сетевой
  вызов через `deps.http_cloud`.
- **Продуктовый трейд-офф (важно):** сейчас погода ВСЕГДА вшита в промпт (нулевая задержка, модель
  всегда «знает» погоду). Как инструмент — модель зовёт по необходимости: чище и без раздувания
  промпта, но +1 round-trip на вопрос о погоде, и модель должна сама решить вызвать. Варианты:
  (а) полностью на инструмент; (б) оставить лёгкую строку в промпте + инструмент для произвольного
  города; (в) сделать инъекцию опциональной флагом. Рекомендую (а) с заметкой про задержку, или (б)
  если «который час/погода» — частый сценарий.

### Календарь — сложнее из-за кредов
Инструменты: `list_events(from, to)`, опц. `create_event(...)`. Главный вопрос — авторизация:
- **CalDAV (user/app-password)** — проще: Nextcloud/iCloud/Fastmail, и Google через app-password.
  Креды (url, user, password) — обычные поля cfg. Рекомендую как первый шаг.
- **Google Calendar API (OAuth2)** — мощнее, но тяжелее: первичный consent-флоу, хранение и refresh
  токена. Токен класть в `data/` (это наше изменяемое хранилище). Заметный объём работы → как
  следующий шаг, не сразу.
- Любой сетевой вызов — через общий http-клиент; ошибки инструмента возвращать строкой модели (как
  `hub.call`), не ронять прогон.

## 7. На что смотреть при реализации (gotchas)
- **Версия SDK (1.27.2):** свериться с точными сигнатурами `FastMCP.list_tools()`/`call_tool()` и
  формой возврата (в разных 1.x `call_tool` отдаёт либо список контента, либо кортеж) — нормализовать
  в `BuiltinMcpSource`.
- **Async/клиенты:** built-in инструменты, ходящие в сеть, используют общие httpx-клиенты из `deps`
  (cloud — проксируемый), не создают свои.
- **Схемы инструментов:** FastMCP генерит `inputSchema` из тайп-хинтов — писать хорошие докстринги,
  это и есть описание инструмента для LLM.
- **Изоляция отказов и таймауты:** один источник (внешний MCP/календарь) не должен вешать весь хаб.
- **Тестируемость:** built-in `FastMCP` тривиально юнит-тестятся в процессе (зовём `list_tools`/
  `call_tool` напрямую, мокаем сеть через respx).
- **Префиксы имён** должны быть стабильны (id инстанса), иначе модель «забудет» инструмент между
  релизами.

---

## 8. Чего НЕ делать
- Не поднимать отдельные процессы/контейнеры под погоду/календарь (подход A) — оверкилл; дверь к этому
  оставляет `ToolSource` (сменить фабрику на `HttpMcpSource`, не трогая остальное).
- Не делать «фейковый» MCP-реестр (подход B) — теряется «настоящий MCP, выносимый наружу».
- Не оставлять погоду спец-кейсом в промпте параллельно с инструментом без явного решения — выбрать
  один из вариантов §6.

---

### Итог
Built-in интеграции = маленькие in-repo `FastMCP`-серверы, дёргаемые в процессе напрямую
(`list_tools`/`call_tool`, без транспорта), спрятанные за единым `ToolSource`; хаб становится
мульти-сорсным с неймспейсингом имён; всё это — мульти-инстансная категория `mcp` в плагинной
конфиг-системе (виды: `http`/`weather`/`calendar`), у каждого своя `ConfigModel` + опц. промпт +
фабрика. Добавить новый built-in MCP — один файл, ноль правок ядра и LLM-цикла. Погода — тривиально
(решить: инструмент vs инъекция в промпт); календарь — начать с CalDAV/app-password, OAuth позже.
