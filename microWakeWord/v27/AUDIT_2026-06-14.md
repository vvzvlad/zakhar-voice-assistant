# Foundation re-validation audit — 2026-06-14

Triggered after the `predict_clip` state-leak nearly shipped a false "short-fire fix". Goal: validate that
all conclusions + prod-v27 stand on correct measurements. No new training. Adversarial subagents + direct
checks. **Several conclusions changed — read the bold items.**

## 1. Eval harness — CLEAN (reproduced)
Independent adversarial audit of `nr2/eval_fresh.py` reproduced every headline number exactly (v27 FRR 19.6%
= 71/362, FAPH per class, shortfire 0.643). Fresh-state truly applied (new `Model` per clip; 16 streaming
state tensors zero-inited by `allocate_tensors`); determinism verified (bit-identical fresh; shared-instance
leaks). FAPH refractory (cd=25 = 750 ms), hours, channel/normalization, window/stride (W5/stride3) all correct.
Threshold `>` vs `>=` boundary never triggers (outputs quantized, never exactly 0.9). **The measurement code
is sound — the problems below are in DATA and ANALYSIS scripts, not the harness.**

## 2. ⚠️ MAJOR — the "honest device-eval" positives are 88% LEAKED; true held-out recall is ~15pp worse
`nr2/pos_eval.txt` (362) splits by filename stem (`nr2/split.py`), so:
- **285/362 (79%) are synthetic yandex clips whose VOICE is also in train** (all 19 eval yx voices ∈ train;
  utterances disjoint → speaker/TTS-engine leakage). FRR on this subset is ~2× lower than on real held-out.
- **+9% (34/77 "section" real-device clips) leak into v27 training via the v8 feature dirs** the sc_lo config
  pulls in (`features_v8_real` w5, `features_v8_device` w12) — 15/31 eval section uuids ∈ v8 positive sources.
  So the "section = held-out, 0 uuid overlap" provenance was FALSE for the shipped model.
- **Truly-clean held-out = only 43/362 (12%)** (real-device, uuid ∉ train ∉ v8). List: `nr2/pos_eval_clean.txt`.

**Honest FRR on the truly-clean held-out (fresh state, cut 0.9, n=43):**
| model | headline (leaky 362) | TRUE held-out (clean 43) | recall |
|-------|---------------------:|-------------------------:|-------:|
| **v27** | 19.6% | **34.9%** [22–50] | ~65% |
| **v16** | 21.3% | **39.5%** [26–54] | ~60% |
| (yx-leaked subset v27) | — | 15.1% | — |
→ The published ~20% FRR is OPTIMISTIC by ~15pp. **True device recall is ~65% (v27) / ~60% (v16), not ~80%.**
All version comparisons, DET curves, and the version_progress numbers were computed on the leaky 88% eval.

## 3. ⚠️ "v27 strictly beats v16" — NOT significant on recall (paired bootstrap, clean held-out)
On the 43 truly-clean clips, paired bootstrap (5000×): FRR diff v27−v16 = **−4.7pp, CI95 [−16.3, +7.0],
P(v27<v16)=0.74**. v27 is *probably* better but NOT significant — the recall edge is within noise on clean
data. v27's real advantage is the **music-FAPH** (2.9 vs 5.8) — but see #4. **The v27-vs-v16 ship rationale
should be stated honestly: similar recall (n too small to separate), v27 better on music false-alarms.**

## 4. Negative-eval leakage (FAPH optimistically biased, small magnitude)
- **Music: 3/40 eval clips byte-identical (same MD5) in v8 training** (`v8/devneg_list.txt`). Removed →
  `nr2/neg_music_eval_clean.txt` (37). music is v27's dominant FAPH class, so re-measure on the clean list.
- silence/music negs are consecutive slices of the SAME continuous room recording (session leakage; FAPH not
  environment-held-out). radio split clean (temporally blocked tv_00863+). short_negs clean (no pos_eval leak).

## 5. 2-stage verifier "51% held-out reject" — REFUTED (was a scaling artifact)
The "embeddings inseparable / coin-flip 51%" came from `nr2/img_embeddings_cloud.py` using raw logreg (NO
feature standardization) + arbitrary 0.5 threshold. With standardization, at matched 95% recall the held-out
FA-reject is **81%** (radio 86%, speech 85%, music 75%), not 51%. The 2-stage dismissal was based on a flawed
number — it does NOT fully revive the option (81% reject still leaves angular's 21.8/h radio at ~4/h, n small),
but "embeddings don't separate" is WRONG and the option deserves re-examination, not dismissal.

## 6. Duration-causality / occlusion — partly overstated + a sample-rate BUG
- ROBUST: a sustained vowel «ааа» with the onset removed scores ~0.033 (genuine onset-dependence) — holds every
  way (cut points, RMS-norm, both audio paths). The model needs the «заха» onset.
- **OVERSTATED:** "lengthening the vowel doesn't change the score / duration doesn't matter" — the model IS
  length-sensitive (trim vowel <50% or speed the word up → score collapses to 0.03). Honest claim: needs onset
  AND a minimum word length.
- **BUG:** `nr2/occlusion_v3.py` + `zakhar_article_images/eval_audit/img_occlusion_final.py` read the silero
  drawn clips (48 kHz) with `sf.read` and feed them to the 16 kHz pipeline (3× dilation, OOD audio). On proper
  16k resample the "informative" occlusion clip flips (ds_901_00282 goes flat; 237/394/313 drop) — so the
  shipped occlusion figure's chosen clip is an artifact of the 48k/16k bug, i.e. cherry-picked by accident.
  The "occlusion is unreliable here (saturation+max-pooling)" self-assessment in VERDICT.md still holds.
- The "сахар→0.999 after захар" anecdote was 1 of 4 speakers; fresh сахар = 0.033 for all.

no operating point reaches deployable FAPH at usable recall (cut0.99 → recall 70% / radio 6.6). Genuinely dead
