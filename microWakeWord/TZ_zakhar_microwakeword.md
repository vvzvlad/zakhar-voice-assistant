# ТЗ: обучение wake word «Захар» для microWakeWord

> Адресат: агент на ноде `agent-node` (10.20.30.226).
> Цель: разобраться с пайплайном microWakeWord и **натренировать собственную on-device
> модель wake word «Захар»** для ESP32-S3 (Home Assistant Voice PE), пригодную для
> компонента ESPHome `micro_wake_word`.

---

## 1. Контекст

Проект `zakhar-voice-assistant` — голосовой ассистент для колонки «HA Voice PE» (ESP32-S3).
Активирующее слово — **«Захар»** (в записях произносится протяжно: «захааар»). Сейчас нужно
получить локальную (on-device) модель распознавания этого wake word, чтобы колонка
просыпалась по слову «Захар» без отправки звука на сервер.

microWakeWord — это TensorFlow-фреймворк (автор Kevin Ahrendt, репозиторий теперь
`OHF-Voice/micro-wake-word`), который выдаёт **квантованную INT8 TFLite-модель** для
TensorFlow Lite for Microcontrollers, плюс **JSON-манифест** для ESPHome. Двухстадийная
детекция: аудио 16 кГц → препроцессор `micro_speech` даёт 40-полосные спектрограммные
признаки каждые 10 мс → потоковая нейросеть (MixConv) выдаёт вероятность → срабатывание
по скользящему окну.

Рядом лежит подробный研究-отчёт `compass_artifact_wf-7bd4ae7f-...md` (он будет загружен
вместе с этим ТЗ) — **прочитай его первым**, в нём весь контекст по сложности, тулингу и
подводным камням. Краткая выжимка ключевого — ниже.

Ресурсов с большим запасом — **не экономь искусственно**: качай полные датасеты негативов,
ставь нормальный batch_size, используй все ядра. Замечания:

- GPU нет → **обучение CPU-only**. Это главный (и единственный серьёзный) ограничитель:
  прогон дольше, но 50 ядер это компенсируют — задействуй многопоточность.
  Долгие прогоны веди в `screen`/фоне с логами.
- Если по ходу дела ресурсов всё же не хватит — запроси у оператора добавку, назвав цифры.
- Питон-зависимости (tensorflow, torch для Piper) ставь в **venv**.
- Docker-тренеры сообщества (`TaterTotterson/...-Nvidia-Docker`) **бесполезны без GPU** —
  иди через нативную установку `OHF-Voice/micro-wake-word` (pip) либо CPU-вариант.
- Python 3.13 может конфликтовать с зависимостями microWakeWord (репо рассчитан на 3.10–3.11).
  Если упрёшься — поставь Python 3.11 (pyenv/uv/deadsnakes-аналог под Debian) в venv.

**Логика по ресурсам:** сначала прикинь, что нужно → запроси недостающее у оператора →
получив апгрейд, тренируй без искусственной экономии. Единственный нерешаемый этим
ограничитель — отсутствие GPU (обучение дольше, но выполнимо на CPU).

## 4. Что нужно сделать (пайплайн)

Разберись по research-отчёту и официальному README, затем выполни:

1. **Подготовь окружение**: clone `https://github.com/OHF-Voice/micro-wake-word`, venv,
   зависимости, swap, проверка диска. (Опорный ноутбук — `notebooks/basic_training_notebook.ipynb`.)
2. **Сгенерируй синтетические позитивы** «Захар» через `piper-sample-generator`
   (`https://github.com/rhasspy/piper-sample-generator`): тысячи TTS-клипов **русским голосом**,
   с вариациями скорости/спикеров. Поэкспериментируй с фонетическим написанием для лучшего
   произношения TTS (например `захар`, `захаар`, `заха́р`) — выбери то, что звучит ближе к
   реальным записям.
3. **Добавь 621 реальную запись** (из §2) в позитивный класс с повышенным весом.
4. **Скачай негативы/фон** (предгенерированные негативные спектрограммы
   `https://huggingface.co/datasets/kahrendt/microwakeword`, при наличии места — RIR/AudioSet/FMA),
   **подмножествами** под лимит диска.
5. **Аугментация** (SpecAugment, фон, реверберация, изменение скорости) и **конвертация в
   40-полосные спектрограммы** (Ragged Mmap). `feature_step_size: 10` (v2-формат — обязателен).
6. **Настрой `training_parameters.yaml`** и запусти обучение через `model_train_eval.py`.
   Выбор лучших весов двухступенчатый: сначала минимизируй `ambient_false_positives_per_hour`
   (FAPH) до порога, затем максимизируй recall.
7. **Конвертируй в квантованный INT8 TFLite + сгенерируй JSON-манифест (version 2)**.
8. **Подбери пороги** `probability_cutoff` и `sliding_window_size` (по дефолту okay_nabu:
   `probability_cutoff: 0.97`, `sliding_window_size: 5`). Прогони итоговую модель на отложенной
   части реальных записей (held-out, не участвовавших в обучении) и оцени recall; оцени FAPH
   на фоновом аудио. Если «срабатывает на всё» — повышай `probability_cutoff`/`negative_class_weight`;
   если «не ловит» — понижай порог / добавляй вес позитивам.

## 5. Ключевые требования и тонкости

- **Формат входа модели**: 16 кГц, моно, 16-bit PCM. Все позитивы (синтетика + реальные)
  привести к нему.
- **Манифест v2**: поля `probability_cutoff`, `sliding_window_size`, `feature_step_size` (=10),
  `tensor_arena_size`, `minimum_esphome_version`, `micro` version 2. Целевой ESPHome ≥ 2024.7.
- **Фраза короткая**: «Захар» — это всего ~2 слога, тогда как research рекомендует 3–4 слога
  для низкого числа ложных срабатываний. Это повышает риск false-accept. Частично это
  компенсируется протяжным произношением «захааар» в записях — **сохрани эту протяжность** в
  синтетике (медленный TTS-вариант) и при подборе порогов будь готов держать `probability_cutoff`
  повыше. Если ложные срабатывания неустранимы — зафиксируй это в отчёте как ограничение фразы.
- **Веса классов**: реальные записи — повышенный вес относительно синтетики; баланс
  `positive_class_weight` / `negative_class_weight` крути для целевого FAPH.
- **Шаги обучения**: ориентир сообщества — 20000–50000+ шагов для нормальной сходимости
  (на CPU это долго — планируй прогон в фоне/`screen`, логируй прогресс). batch_size типично
  128–256; при нехватке RAM снижай.
- **Итеративность**: заложи 2–4 цикла «обучение → тест → тюнинг», а не один прогон.

## 6. Критерии приёмки (Definition of Done)

Сложить артефакты в `/home/claude/zakhar-mww/output/`:

1. **`zakhar.tflite`** — квантованная INT8 потоковая модель.
2. **`zakhar.json`** — корректный манифест **version 2** (с `feature_step_size: 10`,
   подобранными `probability_cutoff` и `sliding_window_size`, валидным `tensor_arena_size`).
3. **`TRAINING_REPORT.md`** — отчёт: какой пайплайн/тулинг выбран и почему; сколько позитивов
   (синтетика + реальные) и негативов; гиперпараметры; **итоговые метрики** (оценка FAPH и
   recall на отложенных реальных записях); финальные `probability_cutoff`/`sliding_window_size`
   и обоснование; известные ограничения (в т.ч. короткая фраза, ресурсы ноды).
4. **ESPHome-сниппет** `micro_wake_word` (YAML) со ссылкой на `zakhar.json` — готовый к вставке.
5. Если что-то из этого недостижимо на текущей ноде — **точный список блокеров** и что нужно
   (диск/RAM/GPU/другая нода), плюс максимально доведённый промежуточный результат.

**Проверка работоспособности**: модель должна загружаться и давать осмысленный отклик —
высокую вероятность на отложенных реальных записях «Захар» и низкую на произвольной речи/фоне.
