# capture_playback.py — сбор «device-real» позитивов через проигрывание

## Зачем это
Voice PE прогоняет звук через **XMOS + TF4micro (NS/AGC)** ДО того, как его увидит
wake-word-модель. Мы же учим на чистом TTS/студийных записях — это **не то**, что модель
реально получает (главный «train/serve gap», см. [DEVICE_CAPTURE_PLAYBOOK.md](DEVICE_CAPTURE_PLAYBOOK.md)).

Скрипт автоматизирует **playback re-recording**: проигрывает синтетические клипы «захааар»
через колонку, а само устройство их записывает (post-frontend PCM, ровно то, что идёт в
модель). Получаем те же клипы, но «пропущенные через реальный аудиотракт» — это и закрывает
разрыв + добавляет реверберацию/далёкое поле. Делается в цикле по папке, без ручной возни.

> Важно: это даёт реализм **канала**, но голоса остаются синтетическими. Диверсити реальных
> людей добавь отдельно — несколькими живыми записями (см. playbook).

## Как работает
Запись идёт не напрямую на устройстве, а через **серверную панель** этого проекта:
`POST http://<host>:8201/api/capture {"device":"voice0a332d","seconds":N}` → возвращает WAV
записи (сервер по ESPHome Native API жмёт на устройстве кнопку capture и стримит звук обратно).

Для каждого клипа из `--src`:
1. считает длительность → окно записи `seconds = ceil(pre_roll + dur + tail)`;
2. в фоне стартует запись на устройстве на `seconds`;
3. через `pre_roll` сек (чтобы устройство успело начать писать) проигрывает клип (`afplay`);
4. сохраняет полученный WAV в `--out/<имя>__dev.wav`.

## Предусловия
- **Сервер запущен и подключён к устройству:** `make run` (панель на `:8201`, устройство
  `voice0a332d` онлайн). Без этого скрипт сразу отвалится с понятной ошибкой.
- Прошивка с capture-сущностями (`capture_sample` / `capture_seconds`).
- macOS с `afplay` (или задай свой плеер через `--player`).
- Колонка, в которую слышно устройство; тихо в комнате во время прогона.

## Запуск
```bash
# из корня репозитория, при запущенном `make run`:
python3 microWakeWord/capture_playback.py                 # все клипы из positive_samples_generated/
python3 microWakeWord/capture_playback.py --limit 5       # сначала проверь на 5 клипах
```
Результат — записи в `microWakeWord/positive_samples_recorded/` (`*__dev.wav`).

## Параметры
| Флаг | Дефолт | Зачем |
|------|--------|-------|
| `--src` | `microWakeWord/positive_samples_generated` | папка с клипами для проигрывания |
| `--out` | `microWakeWord/positive_samples_recorded` | куда писать записи с устройства |
| `--device` | `voice0a332d` | имя ESPHome-устройства (из `.env` ESPHOME_DEVICES) |
| `--panel` | `http://127.0.0.1:8201` | адрес серверной панели |
| `--pre-roll` | `1.0` | пауза после старта записи до проигрывания (даёт устройству начать писать) |
| `--tail` | `1.5` | сколько ещё писать после конца клипа |
| `--gap` | `0.7` | пауза между клипами |
| `--player` | `afplay` | команда плеера (путь к клипу добавляется последним аргументом) |
| `--limit` | — | обработать только первые N клипов |
| `--retries` | `1` | повторы при временных ошибках записи (409 занят/оффлайн, 504 таймаут) |

## Советы по качеству
- **Варьируй** дистанцию (1/3/5 м), громкость, угол, комнату и **источник воспроизведения**
  (ноут / телефон / BT-колонка) между прогонами — чтобы модель не зашила АЧХ одной колонки.
- Если записи обрезают начало слова — увеличь `--pre-roll`.
- Тем же скриптом можно гнать **негативы** (вокальная музыка / речь / «Сахар»-слова): просто
  укажи `--src` на папку с этими клипами и `--out` в отдельную папку.
- Часть записей отложи в held-out для device-eval (в обучение не давать).
