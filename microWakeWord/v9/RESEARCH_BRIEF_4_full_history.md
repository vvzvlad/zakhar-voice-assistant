# Запрос на исследование №4: полная история «захааар» (v1–v9) — что пробовали, что нет, что дальше

## Задача исследования
Дана подробная история разработки on-device wake word. Нужно: критически оценить, **что ещё
стоит попробовать** (особенно свежие/неочевидные техники 2024–2026), как добить открытые
проблемы, и где мы у практического потолка. Ответ — приоритизированный список конкретных
шагов с источниками; явно отметить, что уже исчерпано.

## Что строим и ограничения
On-device wake word — ПРОТЯЖНЫЙ «захааар» (рус., долгая «а»; короткий «захар» НЕ цель) на
microWakeWord (OHF-Voice/micro-wake-word): TensorFlow → квантованный INT8 streaming TFLite для
TFLM на ESP32-S3 (Home Assistant Voice PE) через ESPHome `micro_wake_word`.
Жёсткие рамки: модель ~32k параметров, ~78 КБ INT8, вход 16 кГц моно, 40-полосные micro_speech
спектрограммы (feature_step_size 10), потоковый инференс, tensor_arena ~45 КБ, обучение
CPU-only (ядер/RAM/диска вдоволь, GPU нет). Фраза фиксирована. Девайс имеет XMOS XU316
(AEC/шумоподавление/AGC) + TF4micro ПЕРЕД моделью; есть встроенный VAD-прегейт.

## Текущее лучшее (v8) — продакшен
Обучена на чистых (STT-фильтрованных) синтетических позитивах + РЕАЛЬНЫХ записях, прошедших
аудиотракт устройства (playback re-recording). Оценка на leakage-safe DEVICE-eval (реальный
тракт, 153 held-out позитива + held-out device-шум):
- **on-device FRR 18% (v5) → ~5% (v8)**, recall 0.95 @cutoff0.8; device-FAPH **0/час с VAD**.
- На музыкальном стриме (40 мин) при 0.95/win5: false-fires 0.0/час.
- Деплой: cutoff 0.95, sliding_window_size 5, VAD on.

## ЧТО ПРОБОВАЛИ — и результаты

### Версии моделей
| версия | рычаг | результат |
|--------|-------|-----------|
| v1 | baseline: drawn-out синтетик + 621 real, neg wt5 | рабочая, music-FAPH ~8/ч noVAD / 0 VAD |
| v2 | negative_class_weight 30 | РЕГРЕСС (пере-консервативна, recall упал) |
| v3a/v3b | выделенные муз-негативы; крупнее модель (5-блок 32k) | не побили v1 |
| v4 | масштаб негативов (HF+MUSAN+ESC-50+FMA) + confusables 8/5 + Silero + music-aware отбор | recall 99%, music-FAPH 2.67 noVAD |
| v5 | drawn-out центрирование + adversarial mining раунд-1 + рычаг длительности/окна | «лучшая» до v8 (но eval тёк по uuid) |
| v6/v7 | adversarial mining раунды 2–3 | ПЛАТО, без улучшений |
| **v8** | чистый синтетик + **РЕАЛЬНЫЕ device-капчи** (playback, вес 12) + far-field RIR + uuid-safe split | **продакшен: device-FRR 5%** |

### v9 — программа «вширь» (всё гейтилось на device-eval; ВСЁ, кроме verifier, регресс)
| техника | результат |
|---------|-----------|
| Knowledge distillation (teacher→32k student) | без выигрыша (нужно >400k позитивов) |
| **Second-stage verifier** (65-парам логрег на bottleneck) | **РАБОТАЕТ**: отвергает 100% намайненных no-VAD муз-ложных, 100% истинных wake, ~260 байт, ~0 латентности. Деплоится (не отгружен, но готов) |
| QAT | не нужен — INT8 PTQ уже без потерь (≤0.7pp) |
| Focal loss (γ=2) | регресс |
| margin-BCE (AM-softmax approx, logit-shift) | регресс (НО настоящий angular-margin head не пробован) |
| residual connections | вредят (0.065→0.137) |
| SLR28 RIR-банк | вредит (слишком гулкий; mit_rirs лучше) |
| агрессивная multi-condition аугментация | пере-портит (плато ~20×) |
| арх-свип (BC-ResNet/TC-ResNet) | не побили mixednet |
| софт-микс музыки в обучение | НЕВЕРНОЕ распределение (см. ниже) — вредит |

### Методы/данные (что СРАБОТАЛО)
- **VAD-прегейт** (ESPHome `vad:`): инструментальная музыка → 0 ложных. Дёшево, стандарт.
- **Playback re-recording** (запись синтетики/реальных ЧЕРЕЗ динамик→микрофон устройства): закрыл
  train/serve gap, FRR 18%→5%. ГЛАВНЫЙ рычаг.
- **STT-фильтр синтетики** (Vosk): нашли 62% мусора (Piper denis/dmitri ломались на
  переудлинённых написаниях «захаааар» → говорили другие слова). Починили генерацию (правильное
  написание + length_scale/тайм-стретч). Чистый набор 10246/15500.
- **uuid-leakage fix**: 621 «real» — сегменты с повторяющимися uuid; старый 93-сплит тёк в
  обучение (завышал recall). Починили uuid-disjoint сплитом + device-eval.
- **Длительность как дискриминатор** (протяжное «захааар» + большое окно): режет короткие ложные.

## КЛЮЧЕВЫЕ НАХОДКИ / МЕХАНИЗМЫ
1. **Софт-симуляция канала не работает для barge-in.** Voice PE гоняет AEC с loopback-референсом
   медиа, которое играет САМО → KWS видит ОСТАТОК эха, а не полногромкую музыку. Поэтому
   синтетический music-mix — неверное распределение; реальный остаток снимается только записью
   через устройство (Amazon 1808.00563, Opatka IS-2021).
2. **Ложные в ТИШИНЕ — синтетика их не воспроизводит.** На pure-silence, white/pink room-tone,
   AGC-boosted, syllable-modulated шуме обе модели = 0.0 ложных при всех порогах. Реальные ложные
   в тишине = текстура живой idle-комнаты (гармоники сети 50 Гц, собственный шум микрофона,
   codec-артефакты, AGC-pumping) — нужна реальная запись пустой комнаты.
3. **Модель-сайд исчерпан**: после v8 distillation/QAT/loss/арх дают регресс или 0; прирост
   v5→v8 дали РЕАЛЬНЫЕ данные. ROI сместился на данные + эксплуатацию.
4. Худшие режимы (стратифицированный eval): `music_snr5`, `reverb+music10`, `muffled`.
5. Средние метрики ~на уровне индустрии (Porcupine 97.1%, HA Collective FRR 18%→5%) — близко к
   практическому потолку 2-сложной фразы; выигрыш теперь в WORST-CASE, не в среднем.

## ЧТО ЕЩЁ НЕ ПРОБОВАЛИ / ОТКРЫТО
- **Реальные long-tail device-данные** (планируются): barge-in (через AEC устройства),
  речь+внешнее медиа, far-field >5 м, шёпот/крик, дети/пожилые, **idle-комната/тишина (негатив)**.
- **Настоящий angular-margin head** (L2-norm + additive margin), а не logit-shift аппрокс.
- **On-device персонализация** (few-shot enrollment) — не построена.
- **Production-петля** (логирование эмбеддингов промахов → майнинг hard-neg → переобуч) —
  заскаффолжена, не запущена.
- **Альтернативный фронтенд** (PCEN / ΔLFBE для gain-инвариантности) — micro_speech фиксирован.
- **Больше реальных дикторов** (главный data-рычаг по литературе) — пока десятки.
- **2-я стадия-verifier в проде** — готов, но не задеплоен (т.к. с VAD FAPH уже 0).

## Вопросы исследования (нужны конкретные, источниковые ответы)
1. Учитывая, что синтетика НЕ воспроизводит ни barge-in (AEC-остаток), ни ложные в тишине
   (текстура idle-комнаты) — какие техники, КРОМЕ записи реальных данных, могут это закрыть?
   (например, моделирование AEC-остатка, gain-инвариантные фичи ΔLFBE, halfwave/PCEN, hard-negative
   на собственном шуме устройства). Что реально работает на edge-KWS?
2. Ложные срабатывания на ТИШИНУ/idle у tiny-KWS: известная проблема? Причины (AGC-разгон,
   self-noise, отсутствие silence в негативах) и проверенные фиксы (silence/idle hard-negatives,
   energy/SNR-гейт, спектральный floor-gate, минимальная длительность). Конкретика.
3. Настоящий angular-margin / metric-learning head для streaming KWS под INT8: рецепты,
   ожидаемый прирост против weighted-BCE, как корректно квантовать.
4. Свежие (2024–2026) SOTA для tiny streaming KWS на микроконтроллерах сверх BC-ResNet/MixConv:
   новые архитектуры, self-supervised pretraining, data2vec/wav2vec-дистилляция, что реально
   влезает в ~32k INT8 / 45 КБ arena и превосходит mixednet.
5. Масштаб реальных дикторов: scaling law «дикторы/часы → FRR» для wake word; сколько реально
   нужно для генерализации на незнакомые голоса; эффективность playback-re-recording против
   живого сбора.
6. On-device персонализация (few-shot prototype/verifier) на ESP32-S3: что даёт, как не сломать
   «любой диктор», стоимость.
7. Потолок 2-сложной фразы и приёмлемые цели FRR/FAPH для быта; стоит ли ещё вкладываться в
   модель или ROI полностью в данных/эксплуатации/UX.
8. Production-петля непрерывного улучшения для edge wake word: логирование (эмбеддинги, не звук),
   майнинг hard-neg, критерии переобучения, мониторинг дрейфа — рабочие референсы.
