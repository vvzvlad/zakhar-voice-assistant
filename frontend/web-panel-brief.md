# ТЗ для дизайнера: веб-панель управления Zakhar Voice Assistant

Документ для UX/UI-дизайнера. Описывает, **что** должна уметь панель и **какими настройками**
управляет, не предписывая визуальный стиль. Все названия параметров — реальные, из кода
(`src/settings.py` и др.), чтобы дизайн и бэкенд говорили на одном языке.

---

## 1. Контекст продукта

Zakhar Voice Assistant — серверное приложение, которое заменяет Home Assistant в голосовом
цикле для умной колонки на ESP32 («HA Voice PE»). Сервер подключается к колонке по протоколу
ESPHome (как клиент) и гоняет конвейер:

```
[микрофон колонки] → VAD (определение конца фразы)
   → STT (распознавание речи)
   → LLM (генерация ответа + вызов умного дома через MCP-инструменты)
   → пост-обработка текста (+ опционально RUAccent — расстановка ударений)
   → TTS (синтез речи)
   → [колонка проигрывает аудио по URL]
```

**Пользователь** — владелец/администратор системы (один-несколько человек, технически
подкованные, но хотят не лазить в `.env`). Это **внутренняя админка**, не публичный сервис.
Язык интерфейса — английский/русский.

## 4. Экран «Дашборд / Пайплайн» (центральный)

Главный экран — **горизонтальная схема конвейера** в виде связанных узлов-карточек слева направо:

```
┌──────┐   ┌──────┐   ┌──────┐   ┌───────────┐   ┌──────┐
│ VAD  │ → │ STT  │ → │ LLM  │ → │ RUAccent  │ → │ TTS  │
│      │   │ groq │   │openr.│   │ (выкл/вкл)│   │yandex│
└──────┘   └──────┘   └──────┘   └───────────┘   └──────┘
            whisper-   claude-      опционально     zahar
            v3-turbo   haiku-4.5    перед TTS
```

Требования к узлу-карточке стадии:
- Заголовок стадии + **текущий выбранный провайдер/бэкенд** и ключевой параметр (модель/голос).
- Цветовой статус (настроено и доступно / есть проблема / выключено).
- Клик по карточке → открывает экран настроек этой стадии (раздел 5).
- У RUAccent — явный тумблер «включена/выключена» прямо на карточке (она опциональна).
- Между LLM и MCP — визуальная «ветка»: показать, что LLM ходит в MCP-серверы за инструментами
  (можно бейджем «MCP: N серверов / M инструментов» на карточке LLM, клик → раздел MCP).

Под схемой:
- **Сводный статус системы**: приложение запущено, сколько колонок онлайн, активен ли прокси.
- **Последние запросы** (3–5 строк) с мини-таймлайном стадий и кнопкой «весь лог» (раздел 7).
- Плашка «изменения требуют перезапуска», если есть (см. 2.2).

## 5. Экраны настройки стадий

Общий паттерн экрана стадии: вверху — **селектор провайдера/бэкенда** (сегмент-контрол или
дропдаун), ниже — **набор полей, зависящий от выбранного провайдера** (показываем только
релевантные поля), внизу — кнопка «Проверить» и «Сохранить».

### 5.1. VAD / захват речи

Колонка стримит звук непрерывно и не сообщает, когда фраза закончилась — конец определяем сами
(WebRTC VAD). Это «продвинутый» блок, по умолчанию свёрнут.

| Поле | Параметр | Тип | Дефолт | На что влияет |
|------|----------|-----|--------|---------------|
| Агрессивность фильтра | `VAD_AGGRESSIVENESS` | int 0–3 | 2 | выше = жёстче режет не-речь |
| Тишина в конце (мс) | `VAD_SILENCE_MS` | int | 800 | сколько тишины после речи завершает фразу |
| Мин. речь (мс) | `VAD_MIN_SPEECH_MS` | int | 200 | сколько речи нужно, чтобы «вооружить» детектор конца |
| Макс. длина фразы (мс) | `VAD_MAX_UTTERANCE_MS` | int | 15000 | жёсткий предел — финализируем даже без тишины |
| Таймаут «нет речи» (мс) | `VAD_NO_SPEECH_TIMEOUT_MS` | int | 8000 | если речи вообще не было — завершить |

Подсказать дизайном связь параметров (это «чувствительность к паузам»): возможно, один пресет
«Быстро / Сбалансировано / Терпеливо» + раскрывающиеся точные значения.

### 5.2. STT (распознавание речи)

Селектор **провайдера**: `groq` (облачный Whisper) | `vosk` (офлайн, в процессе).

- При `groq`:
  - Ключ API — `STT_API_KEY` (секрет).
  - Модель — `STT_MODEL` (дефолт `whisper-large-v3-turbo`).
  - (зашито в коде, кандидаты вынести в UI): язык распознавания (сейчас жёстко `ru`),
    `temperature=0`, таймаут запроса 60с.
- При `vosk`:
  - Путь к модели — `VOSK_MODEL_PATH` (дефолт `models/vosk-model-small-ru-0.22`), офлайн, без ключа.
  - Подсказка: модель должна быть скачана на сервере (`make models`).

Кнопка «Проверить» — пробное распознавание короткого сэмпла.

### 5.3. LLM (мозг)

Селектор **провайдера**: `openrouter` | `groq` (оба OpenAI-совместимые chat-completions).

- Ключ — `INTENT_API_KEY` (секрет).
- Модель — `INTENT_MODEL` (дефолт `anthropic/claude-haiku-4.5`; для OpenRouter — слаг провайдера).
- Параметры генерации (сейчас **зашиты**, желательно вынести в UI как «продвинутые»):
  - `temperature` = 0.8
  - `max_tokens` = 4096
  - макс. число раундов «модель↔инструменты» `MAX_TOOL_ROUNDS` = 5 (защита от зацикливания)
  - таймаут запроса = 300с
- Системный промпт и контекст настраиваются в отдельных разделах (4 и 6 ниже по списку) — на этом
  экране дать ссылки-переходы «Редактировать промпт», «Настроить контекст».
- Информационно показать: LLM может вызывать MCP-инструменты (ссылка в раздел MCP).
- Тексты-заглушки (тоже зашиты, можно дать редактировать как «системные ответы»):
  - ответ при rate-limit (429),
  - фолбэк при пустом ответе после инструментов («Готово.»),
  - фолбэк при пустом ответе («Я тебя не расслышала, повтори.»).

### 5.4. RUAccent — расстановка ударений (новая, опциональная стадия)

Стадия **между LLM и TTS**. Сейчас ударения проставляет сама LLM (нотацией «+» перед гласной),
а пост-обработчик переводит их в нужный для движка вид. RUAccent — это автоматический
нейросетевой расстановщик ударений для русского (обрабатывает омографы, «ё»), который снимает
эту нагрузку с LLM и повышает корректность.

Управление:
- **Тумблер вкл/выкл** (по умолчанию выкл).
- Выбор **модели** RUAccent (от лёгких к точным): `tiny` / `turbo` / `big` и пр.
  (точность модели ↔ потребление RAM/латентность — показать как «качество vs скорость»).
- Тумблер «использовать словарь» (точнее, но больше RAM).
- Тумблер «обрабатывать омографы».
- **Пользовательский словарь ударений** — таблица «слово → вариант с ударением» (добавить/удалить).
- Маленькое окно «предпросмотр»: ввести фразу → показать, как проставятся ударения.

Дизайн должен учитывать: стадия влияет на латентность (показать в логе как отдельный отрезок
таймлайна) и на TTS-произношение.

### 5.5. TTS (синтез речи)

Селектор **бэкенда**: `teratts` (HTTP-сервис) | `piper` (офлайн, в процессе) | `yandex`
(Yandex SpeechKit, облако).

Общие:
- Таймаут — `TTS_TIMEOUT` (дефолт 30с).
- Превью: поле ввода фразы + кнопка «Озвучить» с проигрывателем результата.

- При `teratts`:
  - Базовый URL сервиса — `TTS_BASE_URL`.
- При `piper`:
  - Путь к голосу (.onnx) — `PIPER_VOICE_PATH` (дефолт `models/ru_RU-ruslan-medium.onnx`,
    рядом ожидается `<path>.json`).
  - Пауза между предложениями (сек) — `TTS_SENTENCE_SILENCE` (дефолт 0.4).
- При `yandex` (Yandex SpeechKit):
  - Ключ API — `YANDEX_TTS_API_KEY` (секрет; сервисный ключ Cloud, формат `AQVN…`).
  - Голос — `YANDEX_TTS_VOICE` (дефолт `zahar`; дать выпадающий список известных голосов,
    мужские/женские).
  - Эмоция — `YANDEX_TTS_EMOTION` (`neutral` | `good` | `evil`; зависит от голоса).
  - Скорость — `YANDEX_TTS_SPEED` (0.1–3.0, дефолт 1.0; слайдер).
  - Folder ID — `YANDEX_TTS_FOLDER_ID` (нужен только для авторизации от пользователя/IAM;
    для сервисного ключа оставить пустым).
  - Endpoint — `YANDEX_TTS_URL` (продвинутое, обычно не трогают).

## 6. Экран «MCP-серверы» (интеграции умного дома)

LLM управляет умным домом через внешние **MCP-серверы** (например Node-RED), которые
рекламируют инструменты. **Сейчас в коде поддержан один сервер** (`MCP_SMARTHOME_URL` +
опциональный `MCP_SMARTHOME_TOKEN`). **Пожелание заказчика — несколько серверов**, каждый со своим
кастомным промптом. Дизайнить под список (это и архитектурное расширение бэкенда).

Список серверов, у каждого:
- Имя/метка (для человека).
- URL эндпоинта (например `http://10.0.0.5:8001/mcp`).
- Bearer-токен (секрет, опционально; пусто = без авторизации).
- Тумблер «включён».
- Статус подключения + число доступных инструментов + кнопка «Проверить».
- **Кастомный промпт для этого сервера** — текст, который добавляется в системный промпт и
  объясняет модели, что это за инструменты и как ими пользоваться (по запросу заказчика).
- Возможность просмотреть список инструментов, которые сервер реально отдаёт (read-only).

Действия: «Добавить сервер», «Редактировать», «Удалить» (с подтверждением).

Пустое состояние: «Нет MCP-серверов — умный дом недоступен, ассистент только говорит».

## 7. Экран «Системный промпт»

Редактор основного системного промпта (характер «Захара», правила, формат ответов, нотация
ударений «+»).

- Большое поле редактирования (моноширинный, перенос строк), счётчик символов.
- **Плейсхолдер `<<<<<TDW>>>>>`** — на его место подставляются текущие дата/время/погода.
  Дизайн должен показать это как «вставить переменную: дата/время/погода» и пояснить.
- Кнопки: «Сохранить», «Сбросить к шаблону по умолчанию» (с подтверждением), «Предпросмотр»
  (показать промпт с подставленными датой/погодой).
- Источник: файл `data/system_prompt.md`; дефолт берётся из `templates/default_prompt.md`.
- (Параметр пути — `SYSTEM_PROMPT_PATH`, обычно не меняют.)

## 8. Экран «Контекст диалога» (память)

Сколько прошлых реплик помнит ассистент и как быстро «забывает».

| Поле | Параметр | Тип | Дефолт | На что влияет |
|------|----------|-----|--------|---------------|
| Глубина контекста (обменов) | `CONTEXT_MAX_TURNS` | int ≥1 | 5 | сколько последних пар «вопрос-ответ» помнит и шлёт модели |
| Время жизни диалога (сек) | `CONTEXT_TTL_SECONDS` | int ≥0 | 300 | через сколько простоя диалог сбрасывается (0 = всегда с нуля) |
| Папка контекста | `CONTEXT_DIR` | путь | `data` | где лежат файлы контекста по колонкам (обычно не трогают) |

Пояснить дизайном простыми словами: «помнит последние N реплик; если молчали дольше M секунд —
начинает разговор заново». Контекст хранится **отдельно для каждой колонки**.

## 9. Экран «Колонки (устройства)»

Список ESP-колонок, к которым подключается сервер. Формат в коде —
`ESPHOME_DEVICES = name|host|psk;name2|host2|psk2`; в UI это **таблица/карточки**, не строка.

У каждой колонки:
- Имя — `name` (уникальное; по нему же ведётся отдельный контекст).
- Хост/IP — `host`.
- PSK (ключ шифрования ESPHome API) — секрет.
- Статус: онлайн/оффлайн + «Проверить подключение».
- Действия: добавить / редактировать / удалить (с подтверждением).

Общие параметры:
- Порт ESPHome API — `ESPHOME_PORT` (дефолт 6053).
- **`PUBLIC_BASE_URL`** — базовый URL, по которому колонки скачивают сгенерированное аудио
  (например `http://10.0.0.10:8200`). Критичный параметр: если неверный — колонка не проиграет ответ.

## 10. Экран «Лог запросов» (observability) — ключевая фича

Журнал прогонов конвейера с **визуализацией таймингов стадий**. Бэкенд уже измеряет время каждой
стадии (STT, LLM, TTS, общий), поэтому панель должна это показать.

**Список прогонов** (таблица), на строку:
- Время, колонка (имя).
- Распознанный текст (STT) — превью.
- Ответ ассистента (LLM) — превью.
- Общая длительность.
- Итог: успех / ошибка / пустой ввод / причина завершения (`endpoint` / `maxlen` /
  `no_speech` / `device_stop`).
- Мини-таймлайн (полоска со стадиями по длительности).

**Деталь прогона** (раскрытие/дровер) — это самое важное для дизайна:
- **Waterfall/Gantt стадий**: горизонтальные отрезки `VAD захват → STT → LLM → (RUAccent) → TTS`
  с подписями длительности каждой и общей суммой. Видно, где «бутылочное горлышко».
- Внутри LLM — под-шаги: раунды «модель ↔ инструменты», какие **MCP-инструменты** вызывались
  и с какими аргументами, их ответы (это уже логируется).
- Полный распознанный текст и полный ответ.
- Токены LLM (`total_tokens`) и модель, которой реально ответил провайдер.
- Размер аудио и **проигрыватель** синтезированного ответа.
- Ошибки стадий с текстом (если были).

Фильтры: по колонке, по статусу/ошибке, по времени, поиск по тексту.

## 11. Экран «Сеть и интеграции» (advanced)

- **Внешний прокси** — `EXTERNAL_PROXY` (SOCKS/HTTP для облачных API: STT/LLM/погода/Yandex TTS;
  пусто = напрямую). Показать, какие стадии идут через прокси (облачные), а какие — напрямую (локальные).
- **Аудио-сервер** (раздаёт сгенерированное аудио колонкам):
  - `AUDIO_HOST` (дефолт `0.0.0.0`), `AUDIO_PORT` (дефолт 8200), `AUDIO_TTL` (дефолт 300с —
    сколько mp3 живёт в кэше).
- **Погода** (подмешивается в системный промпт):
  - `WEATHER_API_KEY` (секрет, OpenWeatherMap), `WEATHER_CITY` (дефолт `Moscow`).

## 12. Экран «Системное»

- `LOG_LEVEL` (дефолт `INFO`) — уровень логирования.
- Статус приложения (запущено/версия/аптайм), кнопка **«Перезапустить»**.
- Индикатор «есть несохранённые / требующие перезапуска изменения».

## 14. Нефункциональные требования и крайние случаи

- **Безопасность секретов** — маскирование, отсутствие ключей в логах/URL/тостах, HTTPS.
- **Адаптивность** — десктоп в приоритете; 
- **Тёмная тема** — желательна.
- **Состояния**: пустые (нет колонок / нет MCP / нет логов), загрузка, ошибки сети, провайдер
  недоступен, неверный ключ (показать понятное сообщение), несохранённые изменения при уходе со страницы.
- **Крайние случаи логов**: пустой ввод (тишина), ошибка стадии (STT/LLM/TTS упал), rate-limit (429),
  длинный ответ, отсутствие MCP-инструментов.
