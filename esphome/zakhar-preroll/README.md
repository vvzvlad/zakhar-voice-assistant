# voice_assistant с pre-roll — форк для «Захар <команда>» одной фразой

External component для ESPHome, подменяющий встроенный `voice_assistant` версией
с **pre-roll**: устройство постоянно держит кольцевой буфер последних N мс
микрофонного аудио и при старте сессии (например, по `on_wake_word_detected`)
начинает стрим на сервер **с этого буфера**, а не с момента старта. Сервер
получает «Захар <команда>» целиком, говорить можно непрерывно, без ожидания бипа.

База: ESPHome **2026.5.3** (компонент `voice_assistant` скопирован с тега и
пропатчен). Home Assistant Voice PE firmware 26.5.0 требует `min_version: 2026.5.0`,
так что форк совместим со стоковым пакетом. **При апгрейде ESPHome за пределы
2026.5.x форк надо ребейзить** (в dev компонент уже переписан на zero-copy
аудио-источники — диф будет другой).

## Как это работает

Сток: `voice_assistant` создаёт ring buffer (512 мс) только при `start` и сразу
очищает его ПЕРЕД запуском микрофона → аудио до точки детекции не существует.

Форк (активируется опцией `preroll_duration` > 0):

1. **Персистентные буферы.** Ring buffer'ы (512 мс + preroll) аллоцируются в
   `setup()` и никогда не освобождаются (PSRAM, ~64 КБ на канал при 1.5 с).
2. **Пассивные отводы.** Для каждого настроенного канала микрофона codegen
   создаёт второй, *пассивный* `MicrophoneSource` (штатный механизм ESPHome:
   passive-источник не управляет микрофоном, а получает аудио, пока мик запущен
   кем-то другим). На Voice PE `micro_wake_word` держит мик включённым 24/7
   (`stop_after_detection: false` в стоке), поэтому отводы пишут в буферы
   непрерывно; `RingBuffer::write()` затирает старейшие данные → в буфере всегда
   последние `preroll_duration` аудио. Штатные колбэки сессионных источников при
   этом НЕ регистрируются (иначе была бы двойная запись); сессионные источники
   по-прежнему стартуют/стопают мик — это сохраняет ручные сценарии, когда mww
   остановлен (мик поднимется на время сессии, отводы продолжат писать).
3. **Trim вместо reset.** При старте сессии буфер не очищается, а *подрезается*
   до последних `preroll_duration` байт. Стрим в API дренирует буфер от
   старейших данных → сервер получает pre-roll + живой поток одним непрерывным
   аудиопотоком, без склейки и без гонок (в буфер пишет только один таск —
   микрофонный, как и в стоке).
4. **Анти-burst.** Отправка в API ограничена 8 чанками (256 мс) за один проход
   `loop()`, чтобы стартовый залп в несколько секунд аудио не переполнил
   TX-буфер API-соединения.

Единственный producer каждого буфера — колбэк микрофонного таска (как в стоке),
читатель — главный loop. Конкурентный паттерн идентичен стоковому.

## Интеграция в zakhar-voice-preroll.yaml

Положить папку `components/` рядом с `zakhar-voice-preroll.yaml` (или сослаться на git)
и добавить в конфиг:

```yaml
external_components:
  - source:
      type: local
      path: zakhar-preroll/components   # путь к папке components этого форка
    components: [voice_assistant]

# Deep-merge в блок voice_assistant стокового пакета (id: va там уже есть)
voice_assistant:
  preroll_duration: 1500ms
```

Всё остальное (модель «Захар», on_wake_word_detected → voice_assistant.start
из стокового пакета) не меняется: pre-roll прозрачен для автоматизаций.

### Ручной capture без pre-roll

Кнопка «Capture Sample» получит лишние 1.5 с аудио в начале записи.
Если это нежелательно, выключать pre-roll на время capture:

```yaml
button:
  - platform: template
    name: "Capture Sample"
    on_press:
      - micro_wake_word.stop:
      - lambda: id(va).set_preroll_enabled(false);
      - voice_assistant.start:
          silence_detection: false
      - delay: !lambda 'return (int)(id(capture_seconds).state * 1000);'
      - voice_assistant.stop:
      - lambda: id(va).set_preroll_enabled(true);
      - micro_wake_word.start:
```

## Что нужно учесть на сервере

- Аудио сессии теперь начинается за ~`preroll_duration` ДО `run-start`-момента и
  содержит сам wake word «захар». STT-транскрипт будет включать «захар …» —
  сервер должен срезать/игнорировать wake-фразу (поле `wake_word_phrase` в
  `VoiceAssistantRequest` приходит как и раньше). Бонус: сегмент «захар» доступен
  для серверной второй ступени верификации.
- Первые ~1.5 с приходят burst'ом быстрее реального времени — буферизация на
  приёме должна это переживать (обычно тривиально).

## Параметры / ограничения

- `preroll_duration`: 0ms–5000ms (0 = поведение стока, форк полностью пассивен).
  Рекомендация: 1500ms (покрывает «захар» ~0.6–0.8 с + внутреннюю задержку
  детекции micro_wake_word ~0.2–0.5 с + запас).
- RAM: +(preroll+512 мс) × 32 байт/мс × число каналов, в PSRAM
  (~128 КБ при 1500ms и двух каналах Voice PE).
- Если микрофон был остановлен (mute) и сессия стартует в первые
  `preroll_duration` после его включения, в начало может попасть кусок
  «доmute'овой» истории. На практике mww сам требует ~1 с аудио для детекции,
  так что окно нереалистичное.
- Режим `use_wake_word: true` (серверный wake word, непрерывный стрим) с
  pre-roll не тестировался — на Voice PE он не используется (сток: false).
